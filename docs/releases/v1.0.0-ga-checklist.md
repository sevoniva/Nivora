# Future v1.0.0 GA Readiness Checklist

This is a future GA readiness checklist. It is **not** a declaration that Nivora is production-ready or that `v1.0.0` can be tagged today. The current implementation audit classifies Nivora as a beta-candidate foundation. Maintainers should not tag `v1.0.0` until every release blocker is closed or explicitly accepted with a documented limitation.

## API

- [x] API inventory exists in `docs/API_INVENTORY.md`.
- [x] OpenAPI parses through `./scripts/verify-api-specs.sh`.
- [x] Placeholder route groups are documented as `not_implemented`.
- [x] Secret values, token hashes, kubeconfigs, private keys, and credential payloads are not expected in normal responses.
- [ ] Maintainers have reviewed every implemented public route for breaking-change risk.

- [x] Runtime recovery docs exist.
- [x] Event/outbox and reconciliation foundations are documented.

## Known Limitations

- GitOps/Argo CD, cloud inventory, host deployment, DevSecOps external integrations, and web console remain experimental or partial.
- Nivora does not ship a Kubernetes operator.
- Runner sandboxing is not a full untrusted workload isolation guarantee.
- Some persistence and recovery coverage remains beta-level.
- Full enterprise SSO, Vault/KMS lifecycle hardening, SBOM/signing, and external scanner integrations remain future work.

## Release Blockers

- [ ] Any failing `make verify` run.
- [ ] Any committed secret or realistic credential.
- [ ] Any production-insecure default.
- [ ] Any unguarded apply, sync, prune, remote host deploy, Git push, or destructive rollback.
- [ ] Any required production capability marked experimental but advertised as complete or production-ready.
- [ ] Any migration without an up/down pair.
- [ ] Any implemented API/event missing OpenAPI/AsyncAPI coverage.
