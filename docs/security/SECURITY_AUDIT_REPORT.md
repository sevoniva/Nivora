## 3. Operator-Configured Security (Required for Production)

These items require operator action, not code changes:

| Item | Operator Responsibility |
|---|---|
| OIDC Provider | Configure real OIDC IdP (Google, Okta, Azure AD) |
| Runner sandbox | Deploy runners with seccomp/AppArmor profiles |
| TLS termination | Configure ingress/load balancer with TLS |
| Network policies | Restrict pod-to-pod communication |
| Secret rotation | Rotate runner tokens and API tokens regularly |
| Backup encryption | Encrypt PostgreSQL backups at rest |
| Audit log shipping | Export audit records to external SIEM |
| Rate limiting | Configure API rate limiting at ingress |

## 4. Known Limitations (Documented)

1. **Runner is not OS-level sandbox**: Software isolation (workspace, env blocklist, process group). Container/K8s profile requires operator to configure seccomp/AppArmor.
2. **OIDC provider not configured**: JWT validation code exists but no real IdP endpoint configured.
3. **External integrations are skeleton/fake**: Cloud providers, Argo CD, Harbor/Nexus are adapter skeletons.
4. **No WAF/rate limiting**: API does not include built-in rate limiting.
5. **Metrics are in-memory**: Restart resets Prometheus counters.
6. **No mTLS between components**: Server/worker/runner communicate over plain HTTP.

## 6. Audit Verdict

**Hardened beta-candidate: supported for review and controlled validation; production-ready/GA: not passed.**

Nivora has evidence for:
- Access control (RBAC route tests and cross-tenant isolation tests)
- Audit integrity (SHA-256 hash chains, tamper detection)
- Secret management (no leaked secrets, metadata-only storage)
- Production safety gates (config validation, K8s safety policy, artifact immutability)
- Real K8s deployment verified (kind cluster, PostgreSQL runtime)

Remaining items include operator responsibilities and code/validation gaps that must close before production-ready or GA claims.
