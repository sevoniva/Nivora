# Governance

Nivora uses lightweight maintainer-led governance while the project is early.

## Maintainer-Led Early Stage

Maintainers are responsible for keeping architecture boundaries intact, reviewing changes, and preventing phase creep. This is especially important while the domain model, runner protocol, workflow model, and integration model are still forming.

## Proposing Changes

Small changes can start as issues or pull requests. Larger changes should begin with discussion and may require an RFC or ADR.

Architecture-sensitive changes require explicit design review. Examples include runner protocol changes, workflow runtime changes, database model changes, cloud provider adapters, Kubernetes or Argo CD integration design, plugin system changes, security model changes, and public API breaking changes.

## Relationship to ADRs and RFCs

RFCs are proposals. ADRs record accepted architecture decisions. A large RFC may result in one or more ADRs.
