# v1.0.0-rc.1 Readiness Checklist

This checklist is the Phase 9.1 release-candidate hardening baseline. It prepares Nivora for a serious `v1.0.0-rc.1` review, but it is not a production-readiness declaration. No production or GA claim should be made until every release blocker is closed by maintainers.

## Freeze Status

- [x] Feature expansion remains frozen for release-candidate hardening.
- [x] Beta capability boundaries remain documented in `docs/BETA_CAPABILITY_MATRIX.md`.
- [x] API inventory remains documented in `docs/API_INVENTORY.md`.
- [x] RC install, upgrade, backup/restore, troubleshooting, and release automation docs are linked from the documentation index.
- [x] Verification is expected to pass through `make verify`.

## API Stability Review

- [x] Implemented, partial, guarded, and placeholder API groups are inventoried.
- [x] Placeholder APIs must keep structured `not_implemented` responses instead of fake data.
- [x] OpenAPI must parse and cover implemented public routes before RC tagging.
- [x] AsyncAPI must parse and cover emitted events before RC tagging.
- [x] Runner mutation endpoints must remain token-authenticated.
- [x] Credential, secret, token, and runner-token responses must not expose secret values after creation-time one-time returns.
- [ ] Maintainers have reviewed the final RC API surface for breaking-change risk.

## Migration Review

- [x] Migration files live under `internal/infra/migration/`.
- [x] Migrations are paired with up/down files through `000006_performance_indexes`.
- [x] Baseline checks validate migration filename pairing and example references.
- [x] No destructive migration is introduced in this RC hardening pass.
- [ ] Maintainers have run forward/backward migration validation against an isolated PostgreSQL database.
- [ ] Backup has been taken before any migration validation that uses non-disposable data.

## Install Review

- [x] Docker Compose local install path is documented in `docs/operations/install-docker-compose.md`.
- [x] Kubernetes/Helm install path is documented in `docs/operations/install-kubernetes.md`.
- [x] Production-direction install review is documented in `docs/operations/production-install.md`.
- [x] Local binary install review is documented in `docs/operations/production-install.md`.
- [x] Helm template/lint are part of `make verify` when Helm is available.
- [ ] Maintainers have run `make docker-build` in an environment with base-image registry access.
- [ ] Maintainers have run Docker Compose startup and health checks on a disposable local environment.
- [ ] Maintainers have rendered Helm with reviewed values for the intended RC validation environment.

## Runtime And Runner Review

- [x] Runtime recovery operations are documented in `docs/operations/runtime-recovery.md`.
- [x] Runner fleet operations are documented in `docs/operations/runner-fleet.md`.
- [x] Runtime recovery endpoint and reconciliation commands are documented.
- [x] Runner registration, token rotation, heartbeat, and offline detection are documented.
- [ ] Maintainers have run restart/recovery validation with the intended runtime store.
- [ ] Maintainers have validated runner claim/log/status/cancel behavior in an isolated environment.

## Security Review

- [x] Secret scan is required through `./scripts/verify-no-secrets.sh`.
- [x] Credential and secret operations are documented as metadata/reference based.
- [x] Auth/RBAC foundations are documented as beta-candidate foundations, not production SSO.
- [x] Logs, audit records, release notes, and examples must not include raw secrets.
- [x] Runner tokens are stored as hashes and raw tokens are one-time outputs.
- [ ] Maintainers have reviewed auth, RBAC, secrets, logs, and audit behavior before RC tagging.
- [ ] Maintainers have reviewed optional webhook, notification, registry, Argo CD, cloud, and host examples for placeholder-only credentials.

## Performance Smoke

- [x] Performance benchmark and load-script directions are documented in `docs/operations/performance.md`.
- [x] Pagination and API limits are documented as part of the performance foundation.
- [ ] Maintainers have run `make benchmark` on the RC candidate.
- [ ] Maintainers have run optional local load scripts against a disposable server when appropriate.

## Release Automation

- [x] `VERSION` is intentionally left unchanged until the maintainer cuts the RC.
- [x] `CHANGELOG.md` includes a `v1.0.0-rc.1 - Draft` section.
- [x] GitHub release template is RC-aware.
- [x] Release automation guidance is documented in `docs/operations/release-automation.md`.
- [ ] Maintainers have updated `VERSION` to `1.0.0-rc.1` during the actual release cut.
- [ ] Maintainers have created and verified a signed or annotated release tag if required by project policy.

## Release Blockers

- [ ] Any failing `make verify` run.
- [ ] Any committed secret, token, private key, kubeconfig, cloud credential, registry password, or realistic fake credential.
- [ ] Any production-ready, GA-ready, or enterprise-ready claim without maintainer sign-off.
- [ ] Any baseline test requiring Kubernetes, cloud providers, registries, Argo CD, Vault, external scanners, or external notification systems.
- [ ] Any destructive apply, sync, host deploy, rollback, or prune behavior enabled by default.
- [ ] Any implemented public API missing from OpenAPI.
- [ ] Any emitted public event missing from AsyncAPI.
- [ ] Any migration without a matching down migration.

## RC Decision

The RC can be cut only after maintainers complete the unchecked manual review items above and confirm the remaining known limitations are acceptable for `v1.0.0-rc.1`.
